## 1. システム全体における FSM の位置づけ

```
┌─────────────────────────────────────────────────────────────────────┐
│                     運営側プラットフォーム                              │
│                                                                      │
│   G1シミュレータ ←────────────── GEAR-SONIC ←──── 動作DB              │
│        │           トルク/位置指令      ↑                              │
│      音声再生                      動作再生                            │
└────────┬──────────────────────────────┬───────────────────────────────┘
         │ 生成音声 24kHz                │ ZeroMQ
         │ (WebRTC)                     │ ・動作コマンド → ロボット
         │                              │ ・動作実行状態 ← ロボット（実行中/完了）
         │                              │ ・ロボット位置 ← Waypoints
         ▼                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│                      音声対話システム（参加者側）                        │
│                                                                      │
│  ピンマイク音声入力 ──▶ ┌────────────────────────────────────────┐     │
│  ストリーミング 24kHz   │         gpt-4o-realtime                 │     │
│                        │         STT・推論・TTS                   │     │
│                        │  ┌──────────────┐  ┌─────────────────┐  │     │
│                        │  │  対話プロンプト │  │  Function Call  │  │     │
│                        │  │  (FSMが制御)  │  │  ・select_motion│  │     │
│                        │  └──────────────┘  │  ・walk_command  │  │     │
│                        │                    │  ・transit_phase │  │     │
│                        └────────────────────┴─────────────────┘  │     │
│                                    ↑ ↓ プロンプト差し替え/FC結果         │
│                              ┌─────────────┐                           │
│                              │    FSM      │  ← ZMQからの実行完了通知    │
│                              │（フェーズ管理）│  ← タイマー                │
│                              └─────────────┘                           │
└─────────────────────────────────────────────────────────────────────┘
```

FSM は **LLMへ何を指示するか** と **いつ指示を切り替えるか** だけを担う。
発話内容・ジェスチャー選択・アドリブは LLM が担う。

## 2. 状態遷移図

```
              start
                │
                ▼
        ┌───────────────┐
        │   GREETING    │  部長を出迎える
        │   目安 ~30s   │
        └───────┬───────┘
                │ transition_phase("listening")
                │ または 45s 経過（タイマー）
                ▼
        ┌───────────────┐
        │   LISTENING   │  部長の提案を傾聴する
        │   目安 ~90s   │
        └───────┬───────┘
                │ transition_phase("proposal")
                │ または 120s 経過（タイマー）
                ▼
        ┌───────────────┐
        │   PROPOSAL    │  代替案を丁重に提示する
        │   目安 ~120s  │
        └───────┬───────┘
                │ transition_phase("agreement")
                │ または 150s 経過（タイマー）
                │ または 合計 4min 経過
                ▼
        ┌───────────────┐
        │   AGREEMENT   │  合意・締める
        │   目安 ~60s   │
        └───────┬───────┘
                │ transition_phase("closing")
                │ または 60s 経過（タイマー）
                │ または 合計 5min 経過
                ▼
        ┌───────────────┐
        │    CLOSING    │  セッション終了
        └───────────────┘
```

**遷移の優先順位**

```
1位: LLM が transition_phase を呼ぶ  （自然な流れでの移行）
2位: フェーズ上限タイマー            （LLMが移行し忘れた場合の安全網）
3位: 合計経過時間（4min / 5min）     （どのフェーズにいても強制収束）
```

## 6. FSM ↔ LLM 間のメッセージ仕様

### FSM → LLM（2種類）

**① session.update（フェーズ移行時）**
```json
{
  "type": "session.update",
  "session": {
    "instructions": "<base_prompt>\n\n<phase_prompt>"
  }
}
```
フェーズが変わるたびに送る。LLMの「行動指針」が切り替わる。

**② conversation.item.create（残り時間警告）**
```json
{
  "type": "conversation.item.create",
  "item": {
    "type": "message",
    "role": "user",
    "content": [{"type": "input_text", "text": "[システム: 残り60秒です。まとめに入ってください。]"}]
  }
}
```
LLMへの割り込みメッセージ。ユーザ発話として挿入することでLLMが即座に反応する。

### LLM → FSM（1種類）

**transition_phase（Function Call）**
```json
{
  "type": "function",
  "name": "transition_phase",
  "description": "現在フェーズの目的が達成されたと判断したときだけ呼ぶ。フェーズを一段階進める。",
  "parameters": {
    "next_phase": {
      "type": "string",
      "enum": ["listening", "proposal", "agreement", "closing"]
    }
  }
}
```

## 9. 現在の実装上の既知の問題（プロンプトでは解決不可）

### [高] バージイン時に音声が止まらない

**現象：** ユーザが話し始めても、システムの音声がスピーカーから流れ続ける。

**原因：**
```python
# recv_loop 内
elif t == "input_audio_buffer.speech_started":
    self.player.stop()   # MotionPlayer（ジェスチャー）は止まる
                         # _abuf（音声バッファ）は止まらない ← ここが欠落
```

`play_audio()` は `self._abuf` が空になるまで流し続けるが、
`speech_started` 受信時に `_abuf` をクリアする処理がない。

**影響：** ユーザとシステムが同時に発話する。マイクがシステム音声を拾ってフィードバックが起きる。

**修正箇所：** [g1_realtime_dialogue.py:729-731](../mobile-robot-dialogue-system-master_2026-05-26/mobile-robot-dialogue-system-master/src/dialogue_system/g1_realtime_dialogue.py)

```python
# 修正前
elif t == "input_audio_buffer.speech_started":
    print("🎤 [話し中...]")
    self.player.stop()

# 修正後
elif t == "input_audio_buffer.speech_started":
    print("🎤 [話し中...]")
    self._abuf.clear()   # 追加：バッファを捨てて即座に無音にする
    self.player.stop()
```

### [中] VAD の無音判定が早すぎると応答が割り込む

**現象：** 部長が文中で少し間を置いた（800ms 以上）だけで、システムが発話を終わりと判断して応答し始める。

**原因：** VAD の `silence_duration_ms = 800` が短い。会話の「間」と「発話終了」を区別できない。

**現在の設定：**
```python
"turn_detection": {
    "type": "server_vad",
    "threshold": 0.5,
    "silence_duration_ms": 800,   # ← 800ms 無音で発話終端と判定
}
```

**影響：** 部長が考えながら話しているときに被せてしまう。競技評価の (1)ターンテイキングに直結。

**対応方針：** 実機で確認しながら `silence_duration_ms` を 1000〜1200ms に調整する。

---

### [中] select_motion と walk_command が逐次実行になる

**現象：** 「案内しながら歩く」場面で、ジェスチャー完了後に歩行が始まる（同時にならない）。

**原因：** LLM の Function Call は直列に処理される。`select_motion` の FCが完了してから次の `walk_command` の FC が発行される。また `select_motion` の実行コードは動作再生完了まで待機する。

```python
# 現在の実装（_play 内）
self.player.play_once(motion, force=True)
while self.player.is_playing(): time.sleep(0.05)  # ← 完了待ち
self.walker.switch_to_planner()
# ここで初めて walk_command が実行可能になる
```

**影響：** 「こちらへどうぞ」と言いながら歩く自然な動作にならない。評価基準 (4) のエスコート動作の質が落ちる。

**対応方針：** select_motion と walk_command を並列スレッドで同時実行する改修が必要。ただし SONIC のモード切り替え（streaming ↔ planner）との整合性要確認。

---

### [中] 歩行中にTTS音声が再生される

**現象：** `walk_command` の実行中（歩いている最中）に、次の発話音声が流れ始める。

**原因：** TTS 音声の再生（`play_audio`）と歩行コマンドの送信（`walk_command`）は別スレッドで動いており、歩行完了を待たずに音声が流れる。

**影響：** 歩行と発話のタイミングがずれ、「歩きながら説明する」場面の自然さが損なわれる。

**対応方針：** 動作完了フィードバック（ZMQ）が実装された際に、walk 完了後に TTS を解放するゲート機構を追加する（セクション 7 参照）。

---

### [低] ログ出力が斜めになる

**現象：** `run_dialogue.sh` 実行中のターミナル出力が、1行ずつ右にずれていく。

**原因：** `KeyboardController._loop()` 内の `tty.setraw(fd)` が OPOST フラグを無効化し、`\n` が `\r\n` に変換されなくなる。

**影響：** 動作上の問題はないが、ログが読みにくい。デバッグ効率が下がる。

**対応方針：** `tty.setraw` を使わず手動で termios フラグを設定して OPOST だけ残す修正が必要。ただし過去の試みでキーボード制御が壊れたため、慎重に対応する。
