# 起動から対話までの制御フローと対話例

---

## 1. 起動シーケンス

```
ターミナル1: bash scripts/core/run_sim.sh
  └─▶ MuJoCo 起動
  └─▶ ロボットが宙吊り状態で表示される
  └─▶ MuJoCo ウィンドウで「9」キー押下
        └─▶ ロボットが床に着地・立位安定

ターミナル2: bash scripts/core/run_deploy.sh
  └─▶ GEAR-SONIC 起動
  └─▶ WBC 初期化中...
  └─▶ 「Init Done」表示
        └─▶ WBC が「安全モード」（自律立位）で待機中

ターミナル3: bash scripts/core/run_dialogue.sh
  └─▶ shimizu_tencho.yaml 読み込み        ← システムプロンプトをロード
  └─▶ ZMQ ソケット起動 (port 5556)
  └─▶ start_planner() 呼び出し
        └─▶ ZMQ: start=True 送信
              └─▶ WBC が「安全モード」→「SONIC ZMQ制御モード」に切り替え
  └─▶ 2秒待機（WBC安定化）
  └─▶ OpenAI Realtime API に WebSocket 接続
  └─▶ session.update 送信（shimizu_tencho プロンプト + ツール定義）
  └─▶ VAD（音声区間検出）モードで待機開始  ← ここからマイク入力を監視
```

この時点でロボットは **床に立ち、発話待ち状態**。

---

## 2. 一発話あたりの制御フロー

```
ユーザが話す
  │
  ▼ (VAD で発話を検知)
マイク音声 24kHz → Realtime API に WebSocket ストリーミング
  │
  ▼ (STT: Realtime API 内部)
テキストに変換
  │
  ▼ (LLM 推論: gpt-4o-realtime)
システムプロンプト（清水店長キャラ）をもとに返答を生成
  │
  ├─▶ Function Call: select_motion(motion_name="bow_45")
  │     └─▶ [ローカル] SONIC ZMQ制御を一時停止（switch_to_streaming）
  │     └─▶ data/motions/bow_45/sample_1.npz を再生（関節角度列を SONIC に送信）
  │     └─▶ 再生完了後 SONIC ZMQ制御を再開（switch_to_planner）
  │
  ├─▶ Function Call: walk_command(direction="forward", steps=2)
  │     └─▶ [ローカル] walk_forward(2) → SONIC に速度コマンドを送信
  │
  └─▶ TTS 音声 24kHz → WebSocket でストリーミング受信
        └─▶ スピーカーから再生（Jabra デバイス経由）
```

select_motion と発話音声は **ほぼ同時**に始まる（FCコールが発話より若干先）。

## 3. 対話例（5分間シナリオ）

想定状況：
- ロボット = 清水店長
- ユーザ（審査員） = 湯川部長
- 部長が「看板商品ディスプレイを奥の丸テーブル付近に設置したい」と提案してくる

フォーマット：
```
[状態] FSMフェーズ（現在はLLMが判断）
ユーザ:  「発話内容」
  FC:    select_motion(...)  /  walk_command(...)  /  transition_phase(...)
  動作:  [ジェスチャー名] または [移動]
  発話:  「ロボットの返答」
```

---

### フェーズ1: GREETING（0〜30秒）

---

**[0:00] 起動直後（ロボットは待機中）**

```
ユーザ:  「清水店長、本日はよろしくお願いします。」

  FC:    select_motion("bow_45")
  動作:  [45度お辞儀]
  発話:  「湯川部長、ようこそいらっしゃいました。
          お足元の悪い中、ありがとうございます。
          どうぞ、こちらへ。」

  FC:    transition_phase("listening")   ← FSM実装後
```

---

### フェーズ2: LISTENING（〜2分）

---

**[0:30]**

```
ユーザ:  「今日来たのはね、店舗レイアウトについて少し相談があって。」

  FC:    select_motion("nod")
  動作:  [うなずき]
  発話:  「はい、ぜひお聞かせください。」
```

---

**[0:50]**

```

          あの奥の丸テーブルのあたりに、大きなディスプレイを置きたいんだ。」

  FC:    select_motion("lean_forward_interest")
  動作:  [前のめりで聞く姿勢]
  発話:  「なるほど、奥の丸テーブル付近でございますね。
          おっしゃる通り、看板商品の訴求は非常に重要だと思います。」
```

---

**[1:20]**

```
ユーザ:  「そうそう。あそこなら落ち着いてじっくり見てもらえるんじゃないかと思って。
          どう思う？」

  FC:    select_motion("deep_nod")
  動作:  [深くうなずく]
  発話:  「ご説明、よく分かりました。
          じっくりご覧いただける環境、確かに大切ですね。」

```

## 4. 現状と FSM 実装後の違い

| 項目 | 現状（FSMなし） | FSM実装後 |
|------|--------------|----------|
| フェーズ管理 | LLMが自分で判断 | FSMが明示的に管理 |
| プロンプト | 5分間ずっと同じ | フェーズ移行ごとに session.update で差し替え |
| 時間制御 | なし（LLM任せ） | タイマー安全網で強制移行 |
| transition_phase FC | 存在しない | LLMがツールとして呼べる |
| 残り時間警告 | なし | 残り60s/30s で conversation.item.create |

**現状でも**、shimizu_tencho.yaml のプロンプトにフェーズ戦略を全部書いているため、
LLMは上記対話例に近い振る舞いをする。ただし以下が保証されない：
- 傾聴フェーズを省いていきなり反論する可能性がある
- 5分の時間配分をLLMが意識しない
- 残り時間で収束するように誘導できない

## 5. 現状で動かないこと（既知の制約）

| 制約 | 内容 |
|------|------|
| ロボットが先に話せない | VADモードのためユーザ発話を待つ。部長が入室しても自動挨拶はしない |
| select_motion と walk の同時実行 | 現状の実装では select_motion 完了後に walk が始まる（並列ではない） |
| 動作完了フィードバック | ZMQからの完了通知を受け取っていない。walk中に次の発話が始まる可能性 |
